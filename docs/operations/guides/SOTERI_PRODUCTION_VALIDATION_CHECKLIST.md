## Short Internal Map

The production app is one Cloud Run service that serves the dashboard and handles
the small admin/self-scheduling API. The current runtime is no longer the old
algorithmic scheduler path.

Main code paths:

- `app/server/serve.py`: HTTP server and API routes.
- `pipelines/dashboard/build_dashboard.py`: generates `dashboard.html`.
- `pipelines/weather/build_weather.py`: refreshes weather JSON used by the
  dashboard.
- `shared/schedule_store.py`: validates and saves self-scheduling data.
- `shared/gcs.py`: restores/persists runtime state in the production GCS bucket.
- `shared/registry.py`: shared route, backpack, and collector metadata.

Main production state:

- `data/outputs/site/schedule_output.json`: current/future schedule claims plus
  `backpack_status`. Expired claims are pruned from this file.
- `data/runtime/persisted/Walks_Log.txt`: completed walk log. Normal claim and
  unclaim actions should not add completed-walk records here. Completed walks
  reappear in the dashboard after upload/Drive poll rebuilds this log.
- Local previews with GCS disabled may have an empty `Walks_Log.txt`; in that
  case, completed walk cards will be absent, but calendar week navigation should
  still include recent/history windows from schedule/weather metadata.
- `data/runtime/persisted/Recal_Log.txt`: recalibration/runtime log state.
- `data/outputs/site/weather.json`: weather advisory data for the dashboard.
- `data/outputs/site/dashboard.html`: generated dashboard served to users.

Important runtime rule:

```text
self-scheduling claim uniqueness = backpack + date + time-of-day
collector double-booking rule = one collector per date + time-of-day
```

Retired runtime rule:

```text
/api/rerun, /api/rerun/a, and /api/rerun/b are intentionally retired.
They should return 410 Gone and should not be used by active automation.
```

Request-flow quick reference:

- Dashboard load: Cloud Run serves `dashboard.html`, which was generated by
  `pipelines/dashboard/build_dashboard.py` from schedule, weather, walk-log, and
  registry data.
- Claim/unclaim: browser calls `POST /api/schedule/claim` or
  `POST /api/schedule/unclaim`; `app/server/serve.py` delegates schedule
  validation/persistence to `shared/schedule_store.py`; production state should
  land in `schedule_output.json` and then GCS.
- Expired reservations: yesterday and older cannot be newly claimed or edited,
  and old reservations are removed from `schedule_output.json`; uploaded data
  restores the walk through `Walks_Log.txt`.
- Backpack status change: browser calls `POST /api/backpack-status`; the server
  writes the selected holder/location under `schedule_output.json`
  `backpack_status`.
- Force rebuild: browser or automation calls `POST /api/force-rebuild`; the
  server refreshes weather/site outputs and regenerates `dashboard.html`.
- Reminder preview/send: browser or admin script calls
  `/api/notifications/preview` or `/api/notifications/send`; the server loads
  opt-ins from notification preferences and uses SMTP only if production secrets
  are configured.
- Drive/Apps Script: Apps Script should call current rebuild/sync paths, not the
  retired `/api/rerun*` scheduler endpoints.

Why these checks matter:

- deploy check proves the new container and env/secrets are actually running
- rebuild check proves weather + dashboard artifacts can be regenerated
- self-scheduling check proves user claims persist in `schedule_output.json`
- backpack status check proves field logistics state persists under
  `backpack_status`
- retired endpoint check proves no old Apps Script or coworker automation still
  depends on the scheduler-era path
- notification check proves reminders are configured in Cloud Run, not just in a
  local shell

## Self-Scheduling Browser Check

Using the production dashboard:

1. Claim one open slot.
2. Try to claim a past date and confirm the UI/API rejects it.
3. Try to create a conflicting claim for the same backpack/date/time-of-day.
4. Confirm the conflict is rejected.
5. Unclaim/delete the test claim.
6. Refresh the dashboard.
7. Confirm the test claim remains removed.
8. Navigate backward in the calendar after expired claims have been pruned.

Expected result:

- uniqueness is enforced by `backpack + date + tod`
- collector double-booking is rejected for the same `date + tod`
- past dates are not claimable or editable
- claim/unclaim persists through refresh
- no completed-walk entry is added to `Walks_Log.txt` for a claim/unclaim
- past/week-window navigation remains available even if the local walk-log
  mirror is empty; completed walk cards require `Walks_Log.txt` entries
